## Phase 11.5: 意图识别模块优化

### 背景与问题发现

Phase 11 已成功实施独立意图识别服务，MMLU 基准测试 100% 通过。但从 benchmark 结果分析发现以下问题：

1. **类别定义模糊**: `computation` 定义过宽（"逻辑推理、数据处理"），导致 64% 的复杂题被误分为 computation
2. **缺少知识查询类别**: 历史、政治、社会学等知识型问题被错误归入 computation
3. **requires_planning 判断标准缺失**: 模型没有明确依据，只能"瞎猜"
4. **缺少示例**: 没有 few-shot examples，分类边界不清晰
5. **置信度未利用**: 低置信度时没有回退机制
6. **无缓存机制**: 相同 query 重复识别，增加不必要延迟

### 实施方案

#### 1. 优化 INTENT_CLASSIFICATION_PROMPT

**新增类别**:
- `knowledge`: 事实/概念/理论查询（历史、政治、经济、管理、物理、化学、生物等）
  - 例："什么是 Porter 五力"、"维果茨基是谁"、"社会契约论的观点"

**细化定义**:
- `computation`: 从"数学计算、逻辑推理、数据处理"精炼为"纯数学计算、公式求解、数值运算"
- 每个类别增加 2-3 个具体示例和"特征"描述

**明确规则**:
```
requires_planning = true 的情况：
1. 类别是 knowledge 或 multi_step
2. 类别是 complex（无论什么领域）
3. 类别是 computation 但复杂度是 medium 或 complex
4. 涉及专业领域知识（物理、化学、生物、历史、政治、经济、管理等）

requires_planning = false 的情况：
1. 类别是 simple_chat
2. 类别是 computation 且复杂度是 trivial 或 simple（纯数学计算）
3. 类别是 semantic 且复杂度是 trivial 或 simple（简单翻译/总结）
```

#### 2. 添加置信度阈值回退机制

当 `confidence < 0.6` 时强制运行 Planner，确保低置信度情况下的处理可靠性。

```python
if intent and intent.confidence < 0.6:
    print(f"Intent: Low confidence ({intent.confidence:.2f}), running Planner for safety", file=sys.stderr)
    intent.requires_planning = True
```

#### 3. 降低温度值

将温度从 0.1 降至 0.0，提升分类确定性。

## 代码变更摘要

### upa.py - INTENT_CLASSIFICATION_PROMPT (约 220 行)

```python
INTENT_CLASSIFICATION_PROMPT = """
你是一个意图分类器。分析用户查询并输出分类结果。

## 类别定义（互斥，选最匹配的）

- **simple_chat**: 问候、闲聊、感谢
  - 例："你好"、"谢谢"、"早上好"、"Hi"

- **computation**: 纯数学计算、公式求解、数值运算
  - 例："2+2=?"、"求积分∫x²dx"、"解方程 2x+3=7"、"计算组合数 C(5,3)"
  - 特征：有明确数值答案，可通过编程直接计算

- **semantic**: 语言相关的语义任务
  - 例："翻译成英文"、"总结这篇文章"、"分析这句话的情感"
  - 特征：输入输出都是自然语言，不需要外部知识

- **knowledge**: 事实/概念/理论查询（新增）
  - 例："什么是 Porter 五力"、"维果茨基是谁"、"社会契约论的观点"、"文艺复兴的背景"
  - 特征：需要特定领域知识（历史、政治、经济、管理、物理、化学、生物等）

- **multi_step**: 需要多步骤协作或外部工具
  - 例："先搜索 X 概念，然后分析它与 Y 的关系"
  - 特征：明确需要 web_search 或 ask_semantic 工具

- **complex**: 复杂推理，需要深度分析或多步推导
  - 例："证明 X 定理"、"分析 X 现象的多个影响因素"
  - 特征：推理链条长，需要多步逻辑推导

## 复杂度分级
...
"""
```
